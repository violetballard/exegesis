2. **Findings**  
- Vault state corruption without a surviving tmp/backup no longer forces a safe lock or rewrite. `_read_state` only marks `invalid_primary` if the primary file still exists after `_load_payload`, but JSON decode errors call `_quarantine_invalid_state` which moves the file away first, so the flag stays `False`. `create_or_open` then proceeds with `{}` metadata and `is_locked=False`, violating the “project-scoped vault with safe recovery behavior” mandate and Milestone 1’s storage hardening scope. Capture `had_primary` before loading and force lock + rewrite when we detect a deleted/invalid primary with no fallback; add a regression test that a corrupt `.vault_state.json` without backup boots locked. (src/qual/storage/vault.py:25-47, src/qual/storage/vault.py:79-95; PRODUCT_VISION.md:22-25; ROADMAP.md:20-33)  
- Invalid or missing `is_locked` metadata silently unlocks the vault. `raw_state.get("is_locked", False)` coupled with `is_locked = parsed if parsed is not None else False` means any unparseable value (or a missing field when the file was tampered with) defaults to `False`, so a malformed state file leaves the vault unlocked rather than failing safe. Default to `True` whenever parsing fails, treat that as metadata corruption, and rewrite the state accordingly, with unit coverage (e.g., persisted `"maybe"` should keep the vault locked). (src/qual/storage/vault.py:31-44; PRODUCT_VISION.md:22-25; ROADMAP.md:20-33)
